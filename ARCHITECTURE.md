# Архитектура Telegram-монитора

## 1. Принцип работы

Проект работает как короткоживущий пакетный запуск:

```text
GitHub Actions
      |
      v
python -m dividend_monitor.runner
      |
      +--> config/companies.yaml
      +--> config/sources.yaml
      +--> data/state.json
      |
      +--> проверка источников через httpx
      +--> нормализация публикаций
      +--> дедупликация
      +--> отправка новых публикаций в Telegram
      +--> запись state.json
      |
      v
GitHub Actions коммитит изменённый data/state.json
```

Один запуск выполняет полный цикл и завершается. Постоянного процесса, polling и базы данных нет.

## 2. Структура файлов

```text
.
├── src/
│   └── dividend_monitor/
│       ├── config.py          # загрузка YAML и переменных окружения
│       ├── models.py          # Pydantic-модели конфигурации и публикаций
│       ├── runner.py          # полный сценарий одного запуска
│       ├── telegram.py        # Telegram Bot API через httpx
│       ├── deduplication.py   # стабильный ID и проверка истории
│       ├── summarizer.py      # краткое описание из исходного текста
│       ├── storage.py         # чтение и безопасная запись JSON-состояния
│       └── sources/
│           └── base.py        # общий интерфейс источника
├── config/
│   ├── companies.yaml         # список компаний и тикеров
│   └── sources.yaml           # источники; реальные URL пока не добавляются
├── data/
│   └── state.json             # история отправленных публикаций
├── tests/                     # unit-тесты без реальных сетевых запросов
├── .github/
│   └── workflows/
│       └── monitor.yml        # расписание, запуск, commit state.json
├── REQUIREMENTS.md
├── IMPLEMENTATION_PLAN.md
└── ARCHITECTURE.md
```

Назначение модулей намеренно узкое:

- `config.py` ничего не получает из сети;
- `models.py` не содержит бизнес-логики отправки;
- `runner.py` только связывает этапы в правильном порядке;
- `sources/base.py` задаёт единый контракт для будущих RSS/Atom-источников;
- `telegram.py` отвечает только за отправку сообщений и сетевые повторы;
- `deduplication.py` не отправляет сообщения и не изменяет файл;
- `summarizer.py` не использует LLM;
- `storage.py` отвечает только за `data/state.json`.

## 3. Модель одного запуска

`runner.py` получает конфигурацию, загружает состояние и обрабатывает источники независимо друг от друга. Ошибка одного источника записывается в результат проверки и не прерывает остальные источники.

Запуск считается неуспешным, если не выполнен критический этап: повреждено состояние, не удалось отправить обязательное сообщение или не удалось сохранить состояние. Если отдельный источник недоступен, остальные источники продолжают работу, а его ошибка попадает в лог и `source_status`.

## 4. HTTP и повторные попытки

Все HTTP-клиенты используют явные тайм-ауты, например:

- connect timeout: 10 секунд;
- read timeout: 20 секунд;
- общий timeout запроса: не более 30 секунд.

Повторять запрос разрешено только для временных ошибок:

- `httpx.TimeoutException`;
- `httpx.ConnectError`;
- `httpx.ReadError` и другие явно сетевые ошибки чтения;
- HTTP `429`, если сервер вернул `Retry-After` или применима задержка;
- HTTP `500`, `502`, `503`, `504`.

Для MVP достаточно максимум двух повторов с небольшой экспоненциальной задержкой. Ошибки `400`, `401`, `403`, `404` и ошибки валидации ответа повторять нельзя: они требуют исправления конфигурации или источника.

Повторы применяются отдельно к каждому источнику и отдельно к каждому Telegram-сообщению. Они не должны превращаться в бесконечный цикл.

## 5. Контракт источника

`src/dividend_monitor/sources/base.py` определяет простой интерфейс:

```python
class Source:
    def fetch(self) -> list[Publication]:
        """Получить доступные публикации или выбросить ошибку источника."""
```

Реальные адаптеры источников будут добавлены позже. На текущем этапе `sources.yaml` описывает только формат, без настоящих URL.

Каждая нормализованная публикация должна иметь минимум:

```text
company
category
title
description
published_at
url
source_name
importance
source_id
discovered_at
source_type
reliability
```

## 6. Формат `data/state.json`

Состояние — обычный JSON-файл без базы данных. Начальный вариант:

```json
{
  "schema_version": 1,
  "last_successful_check": null,
  "last_daily_summary_date": null,
  "sent_items": [],
  "source_status": {}
}
```

После отправки публикаций:

```json
{
  "schema_version": 1,
  "last_successful_check": "2026-07-13T12:30:00Z",
  "last_daily_summary_date": "2026-07-13",
  "sent_items": [
    {
      "deduplication_id": "sha256:...",
      "company": "Сбербанк",
      "title": "Название публикации",
      "url": "https://example.invalid/item",
      "published_at": "2026-07-13T12:00:00Z",
      "sent_at": "2026-07-13T12:31:04Z"
    }
  ],
  "source_status": {
    "example-source": {
      "last_checked_at": "2026-07-13T12:30:00Z",
      "status": "ok",
      "error": null
    }
  }
}
```

Правила хранения:

- время хранить в UTC в ISO 8601;
- `sent_items` ограничивать по возрасту и/или количеству записей;
- записывать JSON атомарно через временный файл и замену целевого файла;
- не записывать в state токены и секреты;
- добавлять публикацию в `sent_items` только после успешной отправки;
- `last_successful_check` обновлять только после завершения всех критических этапов и сохранения состояния.

## 7. Формат `config/sources.yaml`

Реальные источники пока не подключаются. Файл может содержать пустой список:

```yaml
version: 1
sources: []
```

Целевой формат одной записи:

```yaml
version: 1
sources:
  - id: source-id
    name: Человекочитаемое название
    type: rss
    enabled: false
    url: https://example.invalid/feed.xml
    companies: [SBER]
    categories: [news]
    primary: true
    timeout_seconds: 30
    max_retries: 2
```

Поля:

- `id` — стабильный идентификатор источника;
- `name` — название для логов;
- `type` — тип адаптера, например `rss`; поддержка типов добавляется отдельно;
- `enabled` — позволяет временно отключить источник;
- `url` — URL RSS/Atom или другого поддержанного источника;
- `companies` — тикеры компаний, к которым относится источник;
- `categories` — допустимые категории публикаций;
- `primary` — является ли источник первичным;
- `timeout_seconds` — ограничение HTTP-запроса;
- `max_retries` — число повторов только временных ошибок.

Значения `url`, названия источников и список включённых источников будут заполнены отдельной задачей после проверки реальных официальных каналов.

## 8. Алгоритм дедупликации

Для каждой публикации создаётся стабильный `deduplication_id`:

1. взять GUID или ID записи RSS, если он есть;
2. иначе взять нормализованный canonical URL;
3. если URL нет, собрать нормализованные `company + title + published_at`;
4. вычислить SHA-256 от выбранного значения;
5. добавить префикс `sha256:` и сохранить результат.

Затем:

1. загрузить `sent_items` из `data/state.json`;
2. если ID уже есть, пропустить публикацию;
3. если ID новый, подготовить и отправить сообщение;
4. только после успешного ответа Telegram добавить ID в state;
5. после завершения обработать очистку старых ID.

Один и тот же материал из двух источников может иметь разные исходные GUID. Поэтому для источников с известным canonical URL приоритет имеет нормализованный URL; это уменьшает повторы между лентами. Заголовок сам по себе не является достаточным идентификатором.

## 9. Порядок выполнения workflow

`.github/workflows/monitor.yml` должен работать в таком порядке:

1. запуск по `schedule` примерно каждые 30 минут или через `workflow_dispatch`;
2. установить Python 3.12;
3. установить зависимости проекта;
4. получить checkout с правом записи содержимого;
5. выполнить тесты и проверки Ruff;
6. запустить Python-модуль `dividend_monitor.runner`;
7. скрипт загрузит YAML и `data/state.json`;
8. скрипт последовательно проверит все включённые источники;
9. ошибка одного источника будет изолирована и записана в статус;
10. новые публикации будут отправлены через Telegram Bot API;
11. обновлённое состояние будет записано в `data/state.json`;
12. workflow проверит, изменился ли `data/state.json`;
13. изменённый файл будет закоммичен обратно в репозиторий;
14. workflow завершится.

Для защиты состояния от параллельных записей workflow использует:

```yaml
concurrency:
  group: dividend-monitor-state
  cancel-in-progress: false
```

`cancel-in-progress: false` ставит новый запуск в очередь и не позволяет двум запускам одновременно читать и коммитить один state-файл. Если после checkout возникнет конфликт с новым коммитом, workflow должен завершиться с ошибкой, а не перезаписывать чужое состояние.

## 10. Ограничения архитектуры

Такой подход прост и бесплатен, но не делает отправку Telegram и коммит Git атомарной операцией. Если отправка состоялась, а workflow упал до коммита, следующий запуск может повторить сообщение. Это известное ограничение JSON-state архитектуры; для MVP оно принимается и должно быть покрыто логированием и тестами.

Реальные источники, их URL и адаптеры намеренно не добавляются в рамках текущего этапа.
