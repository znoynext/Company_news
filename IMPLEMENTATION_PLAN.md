# План реализации и оценка реализуемости

## 1. Итоговая оценка

Требования реализуемы для бесплатного MVP при следующих допущениях:

- репозиторий публичный либо доступный лимит GitHub Actions достаточен;
- бот отправляет небольшой объём сообщений в один или несколько чатов;
- компании предоставляют доступные RSS/Atom-ленты или стабильные официальные страницы;
- состояние в GitHub репозитории принимается как простое хранилище, а не как транзакционная база данных;
- частота около 30 минут считается допустимой: GitHub Actions не гарантирует запуск ровно по расписанию.

Архитектура будет пакетной и без сервера: один запуск собирает данные, отправляет уведомления, коммитит JSON и завершается.

## 2. Предлагаемая архитектура

```text
GitHub Actions schedule / workflow_dispatch
                |
                v
       загрузка YAML + JSON state
                |
                v
     HTTP-клиенты источников (httpx)
                |
                v
       RSS/Atom parser (feedparser)
                |
       HTML fallback (BeautifulSoup)
                |
                v
 нормализация -> классификация -> deduplication
                |
        +-------+--------+
        |                |
        v                v
 Telegram Bot API    daily summary
        |                |
        +-------+--------+
                v
         обновление JSON state
                |
                v
        commit через GitHub Actions
```

Компоненты:

- `config/companies.yaml` — компании, тикеры, категории и источники;
- `src/` — загрузка конфигурации, источники, нормализация, правила важности, Telegram-клиент и состояние;
- `state/monitor_state.json` — отправленные идентификаторы и журнал последней успешной проверки;
- `.github/workflows/monitor.yml` — расписание, ручной запуск, секреты, тесты, запуск монитора и сохранение состояния;
- `tests/` — детерминированные unit-тесты без обращения к сети.

## 4. Порядок реализации после утверждения анализа

1. Проверить фактическое содержимое репозитория и существующие правила.
2. Проверить актуальный тикер X5 по официальному источнику и зафиксировать источник проверки.
3. Утвердить перечень бесплатных официальных RSS/Atom-источников для каждой компании.
4. Создать Pydantic-модели и загрузчик `companies.yaml`.
5. Создать адаптеры RSS/Atom и ограниченный HTML fallback.
6. Реализовать нормализацию, категории, уровень важности и стабильный ID публикации.
7. Реализовать JSON state с атомарной записью и защитой от повторной отправки.
8. Реализовать Telegram Bot API через `httpx`.
9. Добавить ежедневную сводку и журнал успешной проверки.
10. Создать workflow с расписанием 30 минут и `workflow_dispatch`.
11. Добавить тесты, Ruff и проверку workflow.
12. Выполнить локальную валидацию и проверить итоговый diff.

## 5. Ограничения GitHub Actions

- `schedule` запускается не обязательно точно в минуту и может задерживаться при нагрузке;
- расписание может быть автоматически отключено в неактивном репозитории;
- GitHub Actions имеет лимиты минут, API-вызовов, артефактов и частоты запусков;
- коммит состояния из workflow требует явного `contents: write` и аккуратной настройки автора коммита;
- параллельные запуски могут породить конфликт состояния, поэтому нужна `concurrency`;
- workflow_dispatch удобен для восстановления и проверки, но не заменяет мониторинг;
- падение после отправки в Telegram и до коммита создаёт риск повторной отправки на следующем запуске.

## 6. Риски хранения состояния

- JSON в Git не является транзакционной базой и может конфликтовать при параллельных запусках;
- Telegram-сообщение нельзя надёжно сделать атомарным вместе с записью Git;
- история может разрастаться, поэтому нужны TTL и ограничение размера;
- откат коммита или ручное редактирование состояния может вернуть старые уведомления;
- запись в репозиторий увеличивает историю Git и потенциально раскрывает метаданные публикаций;
- при отсутствии права записи монитор может получать данные, но не сможет зафиксировать дедупликацию.

Мера для MVP: один workflow с concurrency, коммит только после обработки, ограниченная история, понятный журнал и повторяемая идемпотентная логика. При росте нагрузки JSON потребуется заменить на отдельное бесплатное хранилище, но это будет отдельное архитектурное решение.

## 7. Ограничения автоматического парсинга

- RSS может не содержать полного текста, категории или точной даты;
- разные сайты используют разные форматы дат, часовые пояса и кодировки;
- URL, GUID и заголовки могут меняться, что осложняет дедупликацию;
- HTML-структура официальных сайтов может измениться без предупреждения;
- robots.txt, rate limit, антибот-защита и TLS/сетевые ошибки могут блокировать fallback;
- автоматическая классификация по ключевым словам будет ошибаться на неоднозначных публикациях;
- без LLM нельзя надёжно извлекать сложные финансовые выводы и контекст.

Поэтому первыми источниками должны быть RSS/Atom, а HTML-парсинг — коротким и адресным fallback только для заранее проверенных страниц.

## 8. Что упрощается без AI

- краткое описание берётся из RSS description, metadata или первых предложений текста;
- важность определяется прозрачными правилами и ключевыми словами;
- категория определяется источником, конфигурацией и простыми правилами;
- не выполняется прогноз дивидендов, оценка привлекательности акции или интерпретация отчётности;
- ежедневная сводка будет списком фактов и ссылок, а не аналитическим резюме;
- неоднозначные материалы можно отправлять с нейтральной категорией и средней важностью.

Это соответствует требованию не генерировать неподтверждённые выводы.

## 9. Критерии готовности MVP

- workflow запускается по расписанию и вручную;
- все секреты берутся только из GitHub Actions Secrets;
- новые публикации отправляются один раз при штатном выполнении;
- состояние сохраняется в JSON и содержит время последней успешной проверки;
- ежедневная сводка не дублируется в пределах выбранной даты и часового пояса;
- сообщения содержат все обязательные поля;
- один временно недоступный источник не скрывает общий статус ошибки;
- тесты и Ruff проходят;
- в проекте нет VPS, базы данных, очереди, polling-процесса, LLM и платного API.

## 10. Статус анализа

На этом этапе основной код и workflow не реализуются. Для перехода к реализации нужно подтвердить перечень источников, проверить тикер X5 и согласовать принятие JSON-файла в Git как хранилища состояния.
