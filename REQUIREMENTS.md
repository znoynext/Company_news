# Требования к Telegram-монитору дивидендных акций

## 1. Цель

Создать бесплатный MVP Telegram-бота, который периодически собирает публикации по выбранным российским компаниям, отбрасывает уже отправленные материалы и присылает только новые сообщения. Проект не должен требовать VPS, постоянно работающего процесса или платных API.

## 2. Ограничения стоимости и инфраструктуры

Обязательные ограничения:

- запускать обработку только в GitHub Actions;
- не использовать VPS и постоянно работающий сервер;
- не использовать постоянно работающий Telegram polling-процесс;
- не использовать PostgreSQL, Redis, Celery, Docker и Kubernetes;
- не использовать OpenAI API и платные LLM API;
- не использовать платные финансовые API;
- использовать бесплатные публичные RSS, официальные страницы и иные источники, допускающие автоматический доступ;
- для публичного GitHub-репозитория ориентироваться на бесплатный GitHub Actions; для приватного репозитория отдельно учитывать лимиты бесплатных минут и хранилища.

## 3. Технологии

- Python 3.12;
- GitHub Actions;
- Telegram Bot API через HTTPS;
- `httpx` для HTTP-запросов;
- `feedparser` для RSS/Atom;
- `BeautifulSoup` только там, где RSS и метаданных недостаточно;
- `Pydantic` для конфигурации и нормализации данных;
- стандартные JSON-файлы для состояния;
- `pytest` для тестов;
- Ruff для линтинга и форматирования.

LLM для бесплатного MVP не использовать.

## 4. Расписание и жизненный цикл запуска

Основной GitHub Actions workflow должен:

1. запускаться примерно каждые 30 минут через `schedule`;
2. поддерживать ручной запуск через `workflow_dispatch`;
3. загрузить конфигурацию и состояние;
4. получить новые публикации из настроенных источников;
5. нормализовать и классифицировать публикации;
6. сравнить стабильный идентификатор публикации с историей;
7. отправить в Telegram только новые материалы;
8. сохранить обновлённое состояние в JSON;
9. записать время последней успешной проверки;
10. завершить работу.

Для предотвращения параллельных конфликтов workflow должен иметь GitHub Actions `concurrency` с отменой или очередью повторных запусков. Состояние следует коммитить только после успешной обработки; при ошибке отправки нельзя помечать материал отправленным.

Каждая компания должна позволять задать список RSS/Atom-лент, официальных URL и правила сопоставления публикаций. Источники должны быть первичными или явно отмеченными как вторичные.

## 6. Типы публикаций

MVP должен поддерживать следующие категории:

- новости компаний;
- финансовая отчётность;
- дивидендные события;
- корпоративные события.

Для каждой записи желательно хранить исходную категорию и нормализованную категорию, чтобы правила классификации можно было уточнять без изменения формата состояния.

## 7. Формат Telegram-сообщения

Каждое сообщение обязано содержать:

- компанию;
- категорию;
- заголовок;
- краткое описание;
- дату публикации;
- ссылку на первоисточник;
- уровень важности.

Краткое описание формировать только из подтверждённых данных:

- `description` RSS/Atom;
- metadata страницы;
- первых предложений официальной публикации, если это разрешено источником и технически необходимо;
- с ограничением длины для Telegram;
- без генерации неподтверждённых выводов и инвестиционных рекомендаций.

## 8. Дедупликация и состояние

Состояние хранить в JSON в репозитории. Минимально оно должно содержать:

- идентификаторы или хэши уже отправленных публикаций;
- URL, дату и компанию для диагностики;
- время последней успешной проверки;
- дату последней отправленной ежедневной сводки;
- версию формата состояния.

Стабильный идентификатор выбирать в таком порядке: GUID/ID RSS, canonical URL, затем детерминированный хэш нормализованных полей. Нельзя использовать только заголовок: он может повторяться или изменяться.

Состояние должно иметь ограниченный размер: старые идентификаторы нужно удалять по TTL или по максимальному количеству записей, сохраняя достаточное окно для защиты от повторов.

## 9. Ежедневная сводка

MVP должен отправлять одну ежедневную сводку с публикациями за локальные сутки, используя явно заданный часовой пояс. Повторная отправка за ту же дату запрещена состоянием. Поведение при пропуске запусков и при пустом дне нужно определить тестами; предпочтительно отправлять сводку только после успешной проверки.

## 10. Секреты и настройки

В GitHub Actions Secrets хранить:

- `TELEGRAM_BOT_TOKEN`;
- `TELEGRAM_CHAT_ID`.

В репозитории не хранить токены, пароли и персональные данные. Нужны безопасные таймауты HTTP, ограничение размера ответа, понятная обработка ошибок и отсутствие секретов в логах.

## 11. Качество

- тестировать дедупликацию, классификацию, формирование краткого описания, форматирование сообщений и обновление состояния;
- запускать `pytest` и Ruff в CI;
- обработка одного недоступного источника не должна незаметно приводить к ложному журналу успешной проверки;
- успешность запуска должна означать, что критические этапы выполнены и состояние сохранено.
