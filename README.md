# Dividend Monitor

> Исходное описание репозитория: my bot for news.

Бесплатный пакетный Telegram-монитор публикаций компаний. Каждый запуск выполняет проверку источников, отправляет новые публикации и сохраняет историю в JSON. Постоянного сервера, polling-процесса и базы данных нет.

## Локальная проверка кода

Нужен Python 3.12.

```powershell
python -m venv .venv
.\.venv\Scripts\Activate.ps1
python -m pip install --requirement requirements-dev.lock
python -m pip install --no-deps --no-build-isolation -e .
pytest
ruff check .
```

Локальная проверка нужна только разработчику для тестов и линтера. Рабочий мониторинг, Telegram-отправка, AI и сохранение состояния запускаются исключительно из GitHub Actions — компьютер пользователя включать не нужно.

Подключены реальные RSS/HTML-источники из `config/sources.yaml`, а fixture в `tests/fixtures/news.xml` используется только тестами.

## Тесты и качество

```powershell
pytest
ruff check .
```

Конфигурация компаний находится в `config/companies.yaml`, источников — в `config/sources.yaml`, состояние — в `data/state.json`.

## GitHub Actions usage

### AI-улучшение через GitHub Models

Основной workflow автоматически передаёт временный `GITHUB_TOKEN` в GitHub Models и запрашивает у модели `openai/gpt-4o` короткое русское резюме и уровень важности только для новых публикаций. Это GitHub Models, а не OpenAI API: отдельный AI-ключ, Azure-подписка и банковская карта не нужны. Используется только бесплатная квота GitHub Models. Workflow запрашивает только разрешения `contents: write` (для `data/state.json`) и `models: read` (для AI).

Если GitHub Models недоступен, возвращает ошибку или бесплатный лимит исчерпан, AI отключается до конца текущего запуска. Монитор всё равно отправляет новость с существующим детерминированным резюме и сохраняет состояние. Платный режим и запасной платный провайдер не используются.

При необходимости модель можно поменять только в настройках workflow через переменную `AI_MODEL`; по умолчанию используется вариант выше. Не добавляйте в репозиторий личные токены: в GitHub Actions используется уже созданный секрет `GITHUB_TOKEN`.

Для безопасной локальной проверки используйте `DRY_RUN=true`: новости будут загружены, дедупликация, форматирование и при необходимости AI-анализ выполнятся, но Telegram не получит сообщения и `data/state.json` не будет изменён. Для проверки без сети GitHub Models укажите `AI_ENABLED=true AI_PROVIDER=fake`; это тестовый провайдер, а не рабочий режим.

Для безопасной проверки живого AI без Telegram и изменения `data/state.json` запустите workflow вручную с параметром `verify_ai=true`.

Проект использует два плановых workflow:

- `Dividend monitor` — один job каждый час, на 7-й минуте (`7 * * * *`);
- `Daily portfolio summary` — один job в день в `08:17 UTC`.

Оба workflow работают на Ubuntu, не используют matrix jobs, Docker, artifacts или pip-кэш и завершаются сразу после запуска Python-приложения. Рабочие workflow устанавливают только runtime-зависимости из проекта. Dev-зависимости нужны только workflow `Tests`, который запускается при изменениях кода или вручную.

В конце каждого запуска в логе выводятся:

```text
Duration:
Sources:
New publications:
Telegram messages:
Errors:
```

Чтобы посмотреть в Telegram, как выглядит уже отправленная публикация, запустите основной workflow вручную и включите параметр `send_existing_item`. Будет отправлена самая свежая сохранённая запись; мониторинг источников и `state.json` при этом не изменяются. Если одновременно включены оба тестовых параметра, приоритет имеет `send_existing_item`.

Примерная оценка для месяца из 30 дней: основной workflow — `24 × 30 = 720` запусков, ежедневный — ещё `30`, всего около `750` job-запусков. Если каждый запуск занимает до одной минуты, это примерно `750 runner-minutes` в месяц. Фактический расход зависит от длительности запусков и правил округления GitHub для конкретного типа репозитория и runner.
## Health monitoring

Состояние источников хранится в `data/state.json`. Для каждого активного источника сохраняются последняя попытка проверки, последняя успешная проверка и число последовательных ошибок.

После трёх ошибок подряд бот отправляет одно предупреждение. Повторные запуски не дублируют его. После успешной проверки отправляется одно сообщение о восстановлении, а счётчик ошибок сбрасывается.

Ежедневный workflow также проверяет, не было ли основной проверки более двух часов. Еженедельный отчёт состояния отправляется по понедельникам и содержит количество активных и неработающих источников, последние успешные проверки, число новых публикаций и длительность последнего запуска.
